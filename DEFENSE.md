# 教育培训 Agent v2.0 — 答辩准备手册

---

## 一、开场白（1-2 分钟）

> 各位老师好，我今天的项目是**教育培训 Agent v2.0**。
>
> 市面上的 AI 教育产品，本质上只是一个"问答机器人"——学生问什么答什么，没有记忆，没有教学策略，更不知道学生到底学会了没有。
>
> 我做的不是一个聊天机器人，而是一个**真正意义上的教学智能体**。它能感知学生的状态、制定教学策略、选择合适的教学技能，并且在每次互动后自我评估和优化。
>
> 核心理念可以概括为一句话：**以学生为中心、以状态为驱动、以策略为大脑、以服务为支撑的学习 Agent。**

---

## 二、项目解决了什么问题？（痛点分析）

| 痛点 | 传统 AI 聊天 | 本项目方案 |
|------|-------------|-----------|
| **千人一面** | 所有学生得到相同回答 | 维护学生画像（掌握度/情绪/偏好），个性化教学 |
| **没有教学逻辑** | 只是"问一答一" | Policy Engine 决定教什么 + Skill Engine 决定怎么教 |
| **没有记忆** | 每次对话独立 | 会话记忆 + 长期记忆 + 用户画像，跨会话持续跟踪 |
| **无法衡量效果** | 不知道学生学没学会 | 每次互动后评估，定期生成学习报告和洞察 |
| **幻觉问题** | 纯 LLM 容易编造 | 知识图谱 + 向量检索 + RAG，基于真实教材回答 |
| **决策不透明** | 不知道 AI 为什么这样答 | 全链路可观测日志，每步决策可追溯 |
| **策略固化** | 教学策略写死 | 版本化配置 + 变更日志 + 回滚机制，策略可演化 |

### 3.2 重点模块讲解

#### ① Teaching Agent — 9 步流水线（核心中的核心）

```
用户消息进入
  → ① 意图识别（用户想干什么？）
  → ② 能力路由（该用哪个技能？）
  → ③ 策略选择（难度/方式怎么定？）
  → ④ 上下文组装（把学生状态+记忆+知识拼成 prompt）
  → ⑤ 技能执行（调用具体教学技能生成回复）
  → ⑥ 流式输出（SSE 推送给前端）
  → ⑦ 互动评估（这次教得怎么样？）
  → ⑧ 状态更新（学生的掌握度/情绪变化）
  → ⑨ 学习分析（每 10 次互动生成洞察报告）
```

**答辩话术**：
> "传统聊天机器人是'一问一答'，我的系统是'九步流水线'。每一步都有明确职责，模块之间解耦，可以独立替换和优化。"

#### ② 学生状态模型（State）

这是区别于普通聊天机器人的**核心差异点**：

- **LearningState**：每个知识点的掌握度（0-100）、正确率、错误记录、连续答对/答错次数
- **EmotionState**：信心、挫败感、参与度（通过对话内容推断）
- **BehaviorState**：学习偏好（例子优先/类比优先/图示优先/均衡型）
- **GoalState**：当前学习目标和进度

**答辩话术**：
> "我维护了一个四维的学生画像。当一个学生连续答错 3 题，系统不会继续出更难的题，而是自动降低难度并给出鼓励——这就是状态驱动的价值。"

#### ③ 双路由策略

- **IntentClassifier**：基于 LLM 的意图分类 + 关键词兜底（快而稳）
- **CapabilityRouter**：基于 Function Calling 的结构化路由（准而灵活）
- **UnifiedRouter**：融合两者，取长补短

**答辩话术**：
> "我设计了两套路由机制。纯靠 LLM 分类有时候不稳定，纯靠关键词又太死板。所以用 UnifiedRouter 把两者结合——LLM 优先，关键词兜底，兼顾准确性和鲁棒性。"

#### ④ 知识管理与 RAG

- **知识图谱**：节点（知识点）+ 边（前置/相关/扩展/应用关系）
- **向量检索**：ChromaDB + sentence-transformers 本地嵌入
- **混合检索**：向量相似度 + 图谱关系 + Reranker 重排序
- **文档上传**：PDF/DOCX → 解析 → 分块 → 向量化 → 入库

**答辩话术**：
> "纯靠 LLM 回答会有幻觉问题。我建了一套知识图谱加向量检索的 RAG 管线，让 Agent 的回答基于真实教材内容，而不是凭空编造。"

#### ⑤ 可观测性

每个请求产生完整的 **Observability Trace**，记录：
- 意图分类结果
- 路由决策过程
- 策略选择理由
- 技能执行详情
- 评估打分结果
- 状态变更记录

**答辩话术**：
> "AI 决策往往是黑盒。我做了全链路的可观测性，每个请求的每一步决策都有记录，可以在前端直接查看。这不仅方便调试，也让教学过程可审计、可解释。"

## 四、技术栈选择理由（高频问题）

| 技术选择 | 为什么选它 | 替代方案对比 |
|---------|-----------|-------------|
| **FastAPI** | 原生异步、自动 OpenAPI 文档、Pydantic 校验 | Flask 不原生支持异步；Django 太重 |
| **MongoDB** | 灵活的文档模型，适合学生状态这种多变结构 | MySQL 需要频繁改表；Redis 不适合持久化 |
| **ChromaDB** | 轻量级向量数据库，本地部署，Python 友好 | Milvus/Weaviate 太重；FAISS 缺少持久化 |
| **sentence-transformers** | 本地运行，不依赖外部 API，多语言支持好 | OpenAI Embedding 需要付费且有延迟 |
| **React + Zustand** | 轻量状态管理，比 Redux 简单 | Redux 太重；Context 性能差 |
| **SSE 流式输出** | 比 WebSocket 简单，单向推送足够 | WebSocket 全双工过重；Polling 延迟高 |
| **多模型支持** | 统一抽象层，切换模型零成本 | 绑定单一供应商有风险 |

## 六、答辩常见问题及参考回答

### Q1：你的项目和直接用 ChatGPT 教学有什么区别？

> "三个核心区别：
> 1. **有状态**——ChatGPT 不知道你昨天学了什么、哪里薄弱。我的系统维护了完整的学生画像，包括每个知识点的掌握度、学习偏好、情绪状态。
> 2. **有策略**——ChatGPT 是你问什么答什么。我的系统有 Policy Engine 会根据学生状态主动调整教学策略，比如连续答错自动降难度、检测到挫败感给鼓励。
> 3. **可评估**——ChatGPT 不知道你到底学会了没有。我的系统每次互动后都会评估，定期生成学习报告，有数据支撑教学效果。"

### Q2：意图识别的准确率怎么样？如果识别错了怎么办？

> "我采用了双保险机制。主路径是 LLM 分类，准确率通常在 90% 以上；如果 LLM 返回的结果格式异常或置信度低，会自动降级到关键词匹配兜底。同时 UnifiedRouter 会综合两种结果做最终决策。即使偶尔识别错误，教学技能本身也有容错——比如 quick_explain 作为默认技能，任何输入都能给出合理回复。"

### Q3：学生状态是怎么更新的？准确吗？

> "每次互动结束后，系统会调用 Evaluator 对本轮交互打分，然后根据评分结果更新学生的 LearningState。比如答对了某个知识点，该知识点的 mastery_score 会上调；连续答错 3 次，系统会标记该知识点为薄弱项。情绪状态是通过分析对话内容推断的——比如检测到'我不懂''太难了'这类表达，会提升挫败感指标。当然，这种推断不是 100% 准确，但作为教学策略的参考已经足够有效。"

### Q4：知识图谱是怎么构建的？

> "两种方式：
> 1. **自动种子**：系统首次启动时会根据预设的课程结构自动构建基础知识图谱，包含知识点之间的前置、相关、扩展关系。
> 2. **文档扩展**：教师上传教材文档后，系统会解析文档内容，提取知识点，自动补充到图谱中。
> 图谱的边有四种类型：prerequisite（前置）、related（相关）、extends（扩展）、applies_to（应用），支持图谱遍历检索。"

### Q5：为什么用 SSE 而不是 WebSocket？

> "教学场景的核心需求是服务端向客户端推送流式文本，是典型的单向推送场景。SSE 基于 HTTP，实现简单、天然支持断线重连、不需要额外的协议升级。WebSocket 是全双工通信，适合聊天室、协作编辑这种需要双向实时通信的场景，在这里属于过度设计。"

### Q6：多模型支持是怎么实现的？

> "我设计了一个统一的 AIModelProvider 抽象层。所有模型提供商都实现同一个接口，上层代码完全不感知底层用的是哪个模型。DeepSeek、智谱、OpenAI、通义千问走 OpenAI 兼容协议，用一套代码适配；Claude 用原生 SDK 单独实现。切换模型只需要改一个配置项，零代码改动。还有一个 MockProvider 用于开发调试，不需要真实 API Key。"

### Q7：优化层的 VersionedConfig 是什么？有什么用？

> "教学策略不应该是写死的。我设计了一个版本化配置系统：每次修改教学策略（比如调整难度梯度、修改鼓励阈值），都会生成一个新版本，附带变更日志。如果新策略效果不好，可以一键回滚到任意历史版本。这借鉴了 Git 的版本管理思想，让教学策略可以安全地迭代和实验。"

### Q8：项目有哪些不足？未来怎么改进？

> "目前的不足和改进方向：
> 1. **情绪识别精度**：目前基于关键词匹配，未来可以接入更细粒度的情感分析模型。
> 2. **多模态支持**：目前只支持文本，未来可以加入语音交互、手写识别等。
> 3. **多人协作**：目前是单用户系统，未来可以支持班级管理、教师 Dashboard。
> 4. **自动化程度**：优化层目前是半自动（人工审核），未来可以实现全自动的 A/B 测试和策略优化。
> 5. **评估深度**：目前的评估主要是正确率和互动指标，未来可以加入更深层的学习效果评估（如间隔测试、迁移能力测试）。"

### Q9：系统的性能和并发能力怎么样？

> "后端基于 FastAPI 异步框架，数据库用 Motor 异步驱动，天然支持高并发。AI 调用是主要瓶颈，通过流式输出可以极大改善用户感知延迟——用户不需要等完整回复生成，边生成边展示。MongoDB 的读写性能对于教育场景绰绰有余。如果需要进一步扩展，可以引入消息队列做异步任务处理。"

### Q10：测试覆盖情况如何？

> "项目有 22 个测试文件，覆盖了核心模块：模型层、记忆系统、检索层、评估系统、学习分析、优化层、API 端到端测试和集成测试。使用 pytest + pytest-asyncio，支持异步测试。"

## 八、项目亮点总结（一句话版，适合被问到"你的创新点是什么"）

1. **不是聊天机器人，是教学智能体**——有状态、有策略、有评估的完整教学闭环
2. **四维学生画像**——学习掌握度 + 情绪状态 + 行为偏好 + 目标进度，驱动个性化教学
3. **双路由策略**——LLM 分类 + 关键词兜底，兼顾准确性和鲁棒性
4. **知识图谱 + RAG**——解决 LLM 幻觉问题，回答基于真实教材
5. **全链路可观测**——每步决策可追溯，教学过程可审计
6. **多模型统一抽象**——一套代码适配 5+ 个 AI 供应商，零成本切换
7. **版本化教学策略**——借鉴 Git 思想，策略可迭代、可回滚
8. **教学技能自动发现**——插件化注册机制，新增技能零配置

## 九、答辩注意事项

1. **先讲问题再讲方案**——老师更关心"为什么做"而不是"怎么做"
2. **Demo 准备好备用数据**——万一网络不好，提前录好演示视频
3. **诚实说不足**——不要回避问题，主动说不足 + 改进方向更显成熟
4. **准备一张架构图**——答辩最关键的视觉元素，建议手画或用 draw.io
5. **控制时间**——超时是大忌，提前演练 2-3 遍
6. **熟悉代码细节**——老师可能会追问某个函数的实现，要能定位到具体文件和行号
